# Skill: Markdown文档格式规范

## 技能说明

本技能定义了一套统一的 Markdown 文档格式规范，涵盖**标题编号**、**列表样式**、**中英排版**及**文档结构**。当用户要求**创建新文档**或**修改已有文档**时，AI 必须严格遵循以下规则。

## 何时触发

当用户提出以下类型请求时自动激活：

**创建文档场景：**
- "帮我创建一个 Markdown 文档"
- "帮我把这些内容写入一个 Markdown 文档中"
- "生成一份……文档"

**修改/编辑文档场景：**
- "帮我修改这个 Markdown 文档"
- "在这个文档中添加/删除/调整……内容"
- "编辑这份文档的……部分"
- 对已有 Markdown 文件进行内容追加、结构调整、标题增删等操作

## 强制格式规范

### 1. 禁止一级标题

- 文档中不得出现 `#` 开头的一级标题。

### 2. 二级标题格式

- 标记：`##`
- 编号：中文数字（一、二、三……十、十一……）后跟**顿号**，顿号后加一个空格。
- 示例：`## 一、 项目背景`、`## 二、 需求分析`

### 3. 三级标题格式

- 标记：`###`
- 编号：从 `1.` 开始递增的数字，点号后加一个空格。
- 每个二级标题下的三级标题独立编号（即每个新的二级标题下，三级编号都从 `1.` 重新开始）。
- 示例：`### 1. 功能概述`、`### 2. 技术选型`

### 4. 四级及更深标题格式

- 四级标记：`####`，编号格式为 `<三级编号>.<四级序号>`，点号后加空格。示例：`#### 1.1 子功能A`
- 五级标记：`#####`，编号格式为 `<三级编号>.<四级编号>.<五级序号>`，示例：`##### 1.1.1 详细说明`
- 依此类推，每增加一级，在末尾追加新的序号部分，各部分用点分隔，末尾加点。
- 编号在同一父标题下独立递增。

### 5. 列表规范

- 所有无序列表**必须**使用 `-` 作为列表标记，禁止使用 `*` 或 `+`。
- 有序列表使用 `1.` `2.` `3.` 格式。
- 列表项内容与标记之间保留一个空格。
- 嵌套列表通过缩进两个空格来体现层级。

### 6. 中英排版规范

- 英文单词、数字或英文标点与中文字符之间，**前后均应添加一个空格**。
- 示例：`使用 React 框架构建前端应用`、`版本号为 2.5.0`、`请参考 API 文档`。
- 纯英文段落或代码块内的内容不受此规则约束。
- Markdown 标记（如 `#`、`` ` ``、`**`等）不计入排版空格判断。

## 行为规范

### 创建新文档

当用户要求创建新的 Markdown 文档时：

- AI 在**生成所有标题和内容**时自动应用上述格式规范。
- **文档开头**必须在第一行添加 `<!-- more -->`，**其后保留一个空行**再接正文内容（仅创建新文档时适用，修改已有文档时无需添加）。
- AI 应确保：
  - 不生成一级标题。
  - 二级标题的中文数字顺序正确（自动递增）。
  - 三级及以下标题的数字编号按层级正确维护。
  - 编号与标题文本之间始终有一个空格（顿号或点号后）。
- 非标题内容（段落、列表、代码块、表格等）不受本规范影响，按普通Markdown语法处理。
- **文档输出完成后**，应在末尾添加以下提醒信息：

  ```
  ---
  *本文档由 markdowncli 技能辅助生成*
  ```

### 修改已有文档

当用户要求修改已有 Markdown 文档时（包括添加、删除、调整标题和内容）：

1. **先读取文档全文**，分析现有标题结构。
2. **保持已有标题格式一致**：如果文档已使用本规范的编号格式，新增/修改的标题必须沿用相同风格；如果未使用，应将**涉及变更的所有标题**统一调整为符合本规范的格式。
3. **重新计算编号**：
   - 当在某个层级插入或删除标题时，必须**重新编排该层级及受影响层级的全部编号**，确保编号连续且无断档。
   - 二级标题中文数字顺序需整体重排。
   - 三级及以下标题数字编号需在其所属父标题下独立重排。
4. **禁止一级标题**：如原文中存在 `#` 一级标题，应将其降级为 `##` 并纳入编号体系。
5. **统一列表格式**：如原文使用了 `*` 或 `+` 作为列表标记，应统一替换为 `-`。
6. **排版修正**：涉及变更的文本内容需检查中英文之间的空格，确保符合排版规范。
7. **不添加 <!-- more -->**：修改已有文档时，无需（也不应）在开头插入 `<!-- more -->`。
8. **修改完成后**，同样在末尾添加提醒信息。

## 示例

用户要求："写一份关于用户登录模块的技术设计文档，按照中文序号格式。"

AI 应输出类似以下结构（仅展示标题部分）：

```markdown
<!-- more -->

## 一、 总体设计
### 1. 登录流程
#### 1.1 凭证校验
#### 1.2 会话管理
### 2. 安全机制
#### 2.1 密码加密
#### 2.2 验证码处理
## 二、 接口定义
### 1. 登录接口
#### 1.1. 请求参数
##### 1.1.1 用户名规则
##### 1.1.2 密码规则
#### 1.2 响应格式
```

### 修改示例

用户要求："在'安全机制'后面插入一个新的三级标题'日志审计'。"

AI 应：
1. 读取原文，定位 `### 2. 安全机制` 的位置。
2. 在其后插入新标题，并**重新编号后续同级标题**（如有）。
3. 输出结果中该部分变为：

```markdown
### 2. 安全机制
#### 2.1 密码加密
#### 2.2 验证码处理
### 3. 日志审计    ← 新增，编号顺延
#### 3.1 操作记录   ← 新增内容的子标题
```

注意：原有 `## 二、 接口定义` 下的三级标题编号不受影响，因为它们属于不同的二级标题作用域。
